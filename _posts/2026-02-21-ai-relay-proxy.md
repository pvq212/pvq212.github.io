## 前言

這兩年只要聊到 AI API，聊著聊著幾乎一定會有人拋出一句：「要不要走中轉站？」

問題是，很多人嘴裡講的明明是同一個詞，腦中想的卻完全不是同一件事。

例如：

- 官方 API
- 正規聚合層
- 自建中轉面板
- 逆向反代 / Cookie 渠道

這四種東西，風險、穩定性、合規性，差得真的很遠。你如果一開始沒分清楚，後面不是錢白花，就是帳號莫名其妙出事。

## 先給新手一個短答案

### 最穩：官方 API / 官方雲

像 `OpenAI`、`Anthropic`、`Vertex AI`、`Azure`、`Bedrock` 這種。

### 再來才是正規聚合 / gateway

像 `OpenRouter`、`Cloudflare AI Gateway`、`LiteLLM` 這種，但它們也不是完全同類：

- `OpenRouter` 比較像公開聚合入口
- `Cloudflare AI Gateway` 偏向閘道、觀測、控流
- `LiteLLM` 很多人拿來自建自己的 gateway

我把它們放同一層，不是說它們完全一樣，而是它們至少都不是逆向來源，而是比較正規的 gateway / aggregation 類解法。[1][2]

### 工具型：One API / New API 這類面板

工具本身其實中性，真正關鍵的是你把它接去哪個上游。[3][4]

### 最不建議：逆向 / 網頁反代 / Cookie 渠道

這種通常最便宜，也最容易翻車。

## 為什麼我會特別強調逆向風險

因為很多人一看到「便宜」或「免 key」，警戒心就先放下來了。等真的用一陣子，才發現麻煩根本不在功能，而在**這條路到底能不能活久一點**。

我去看現在官方條款，像 OpenAI 直接在條款裡寫了不能 reverse engineer、不能繞過限制與保護機制；Anthropic 也明寫不得 reverse engineer、未經批准不得 resell。[5][6]

這些條款不是寫著好看的，真的會變成帳號風險。

## 中轉面板本身有沒有問題？

這題得拆開看。

像 `One API`、`New API` 這類專案，本質上比較像是統一不同上游 API、做 key 分發、做 gateway 的工具。[3][4]

面板本身不是問題核心。

但如果你拿它接的是來路不明的逆向渠道，那風險還是在。

所以真正該問的不是「這個面板叫什麼名字」，而是：

> 你的上游到底是不是官方授權、是不是穩定、是不是說得清楚。

如果你是第一次自己搭，我會很保守地走：

1. 一個官方上游
2. 一個面板或 gateway
3. 先打通一條流程

不要第一天就接四五家來路各不相同的來源。真的出狀況時，你連該懷疑誰都不知道。

## 如果你真的要買中轉站，我會先問這幾題

老實說，這段比「哪家便宜」重要太多了。

1. 上游是官方 API，還是逆向來源？
2. 壞掉時你能不能知道是哪一層壞？
3. 有沒有清楚的模型、費率、限速、退款與資料處理說明？
4. 你的 key、對話內容、日誌到底經過幾層？

這幾題只要對方講不清楚，我通常就不太敢把正式流量放上去。

## 自建面板跟買第三方入口，差在哪

### 自建面板

像 `One API`、`New API`、`LiteLLM` 這種，比較像你自己拿回控制權。

優點是：

- 上游你自己決定
- 可以自己控流、分發 key、做配額
- 比較知道資料流怎麼走

缺點也很直接：

- 你要自己維護
- 爆了得自己修
- 權限、日誌、金鑰管理都要自己顧

### 買第三方中轉站

優點當然是快，缺點就是黑盒感往往更重。

如果你的需求只是自己小量測試，偶爾也許可以接受；但如果是正式產品，我會很在意它到底是不是把風險全包給你。

## Google Cloud 額度這種東西，最容易被誤解

這段我很想單獨拉出來講。

很多人會說：「Google Cloud 不是有 300 美元免費額度嗎？」

有，但你要看官方文件怎麼寫。

我查現在官方文件，`Google Cloud Free Program` 的確寫新用戶有 `$300 / 90 天`，而且不會自動扣費；但同時也明寫了，這個 free trial **不能直接拿去付 Gemini Developer API (AI Studio) 的成本**。[7]

所以你如果以為 Google 的 300 額度等於 Gemini 全都能直接拿來刷，這個理解就是錯的。

這也是我對某些中轉站宣傳很反感的地方：

它會把「有額度」直接講成「都能拿來跑所有模型」，新手如果沒有自己去查官方文件，很容易就被帶著走。

## 正規聚合層到底值不值得用

我的答案是：值得，而且很多時候比你每家都自己接還合理。

像 `OpenRouter` 這類服務，重點不是模型比較便宜，而是：

- 統一入口
- routing
- fallback
- provider 選擇

這些功能在你開始玩多模型後，真的很省事。[1]

但你也要承認，它畢竟多包了一層。資料流、依賴風險、哪天它自己掛掉怎麼辦，這些都得一起想。

## 如果我是新手，我會怎麼選

### 第一階段：先走官方或正規聚合

不要先碰來源不明的逆向渠道。

### 第二階段：如果要自己整合很多 key，再看 One API / New API 這類面板

### 第三階段：只有在你很清楚風險，而且能承受失效與封禁風險時，才去碰灰色 reverse 方案

## 我認為最實際的底線

如果這條服務：

- 不願意講上游
- 不願意講條款風險
- 不願意講資料流
- 只有「便宜」一個賣點

那我大概率不碰。

因為真正踩坑之後你會發現，最貴的不是單價，而是某天整套流程突然失效，然後你還不知道該怪誰。

如果你只是個人開發者，我真的不太建議把中轉站這件事想成「哪裡最便宜」。

更值得問的是：

- 穩不穩
- 合不合規
- 資料怎麼流
- 哪天掛了會不會整套一起死

便宜當然重要，可是能不能活久一點，通常更重要。
