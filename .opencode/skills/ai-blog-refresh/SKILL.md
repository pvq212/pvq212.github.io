# AI Blog Refresh

把「抓新資料 → 改寫成作者口氣 → 自評 AI 味 → 補強到能發文」這整套流程固定下來。

## 何時使用

當你要為技術部落格撰寫或重寫文章，而且同時滿足下面幾點時：

1. 需要先從最新官方文件、release note、新聞或 GitHub 資料抓資料
2. 文章不能只是整理資料，還要像作者本人寫的
3. 讀者以新手為主，需要兼具教學、介紹、個人感想
4. 需要自己檢查 AI 味、資訊密度、實用性

## 工作流程

### 1. 先盤點作者既有口氣

- 先找作者最像「本人寫的」3 篇文章當 baseline
- 抓出這幾個特徵：
  - 開頭是不是直接切入
  - 有沒有先講自己的場景
  - 有沒有用「這邊、接下來、最後、我自己測試」這種口氣
  - 有沒有實際指令、檔名、路徑、踩坑提醒

### 2. 再抓資料，不要先開寫

資料來源優先順序：

1. 官方文件
2. 官方 release / changelog
3. 官方 GitHub repo / README
4. 官方定價 / 條款 / 支援區域頁
5. 補充性的社群資料或評測

寫作前至少要回答：

- 這工具/服務現在還在積極維護嗎？
- 最近一年有沒有明顯更新？
- 有沒有新手容易踩的坑？
- 有沒有條款、區域、費率、穩定性上的限制？

### 3. 用新手視角整理文章結構

每篇文章至少要有這幾段：

1. 前言：用作者自己的場景切進去
2. 先講結論：適合誰、不適合誰
3. 基本介紹：不要只貼官方定義，要轉成人話
4. 實際做法：指令、設定、路徑、步驟
5. 踩坑提醒：哪裡最容易翻車
6. 個人感想：優點、缺點、是否值得用
7. 讀者角度評分：新手友善度、實作價值、內容新鮮度、AI 味
8. 參考資料：放官方與高可信來源

### 4. 強制做 AI 味自評

初稿完成後，逐條檢查：

- 開頭是不是太像模板文？
- 有沒有太多「不是 X，而是 Y」這類機械句型？
- 有沒有抽象形容太多、具體細節太少？
- 有沒有真的像作者自己用過，而不是只把官網翻成中文？
- 看完後，新手知不知道下一步要做什麼？

如果下面任一項低於 4 分，就重寫：

- 新手友善度（1-5）
- 實作參考價值（1-5）
- 內容新鮮度（1-5）
- 作者感（1-5）

### 5. 圖片與排版規則

- 圖片放在 `assets/blog/<post-slug>/`
- front matter banner 用：

```yaml
image:
  path: /<post-slug>/banner.svg
```

- 內文圖片也用短路徑：

```md
![說明圖](/<post-slug>/1.png)
```

- 參考資料盡量放在文末，用數字參考連結：

```md
[1]: https://example.com
```

### 6. 發文前最後檢查

- front matter 完整
- tags 全小寫
- 內文沒有過度擬人化到像廣告文
- 有明確優缺點，不是單向吹捧
- 有給新手可執行的起手式
- 有資料來源

## 寫作口氣備忘

- 先講自己怎麼遇到這東西
- 多用「這邊、接下來、我自己、先、最後」
- 不要裝全知，該講限制就講限制
- 與其講大詞，不如講真實場景
- 能一句講清楚就不要堆太多漂亮形容詞
